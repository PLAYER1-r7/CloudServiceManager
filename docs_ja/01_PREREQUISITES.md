## **🔐 認証・セキュリティ方針**

### クラウドプロバイダー認証方法
| プロバイダー | 認証方法 |
|------------|---------|
| **AWS** | `~/.aws/credentials` または環境変数 (`AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`) |
| **GCP** | サービスアカウントJSON + 環境変数 (`GOOGLE_APPLICATION_CREDENTIALS`) |
| **Azure** | Azure CLI (`az login`) または環境変数 (`AZURE_SUBSCRIPTION_ID`, etc.) |

### セキュリティ要件
- **認証情報は環境変数または標準の認証情報ファイルから読み込む**
- **ログに認証情報を出力しない**
- **`.gitignore` で認証情報ファイルを除外**

### 認証リクエストルール（必須）
クラウドプロバイダーまたはGitHubの操作に対して認証が必要になった場合：

1. **必ずユーザーに認証資格情報をリクエストすること**:
   - 認証をバイパスするような試みをしないこと
   - プレースホルダー/ダミー認証情報を使用しないこと
   - デフォルト認証情報をハードコードしないこと
   - 認証情報について推測しないこと

2. **必要な認証情報に含まれるもの**:
   - クラウドプロバイダーの認証情報（AWS、GCP、Azure）
   - GitHub認証（Personal Access Token または OAuth）
   - その他のサービス固有のAPIキーやシークレット

3. **認証が必要な場合のプロセス**:
   - 実行を停止し、認証情報を明確にリクエストすること
   - どのサービスが認証を必要としているかを指定すること
   - 認証情報が利用できない場合は、セットアップ手順を提供すること
   - ユーザーが本物の認証情報を提供するのを待つこと

**ルール**: 「認証が必要な場合は、常に明示的にリクエストすること。代替手段や認証回避を試みないこと。」

---

### GitHub 認証ルール（必須）

**GitHub CLI (`gh`) がGitHub操作のための唯一の許可された認証方法です。**

#### 許可される方法
✅ **GitHub CLI (`gh`)** - 唯一の許可された方法
- インストール: `brew install gh` (macOS) または `sudo apt install gh` (Linux)
- ログイン: `gh auth login`
- 自動トークン管理、OAuthベースの認証
- 用途: リポジトリのセットアップ、Issue作成、PR管理、自動化

✅ **GitHub Actions 組み込みトークン** - CI/CDのみ
- GitHub Actionsワークフローで `${{ secrets.GITHUB_TOKEN }}` を使用
- GitHub Actionsによって自動的に提供される
- CI/CD自動化コンテキストでのみ使用可能

#### 禁止される方法
❌ **Personal Access Token (PAT) - Classic** - 禁止
- 古いセキュリティモデル
- 過度に広範な権限
- 手動トークン管理が必要
- GitHubによって廃止予定

❌ **Personal Access Token (PAT) - Fine-grained** - 禁止
- Classic PATより安全だが、依然として手動作成が必要
- 手動の期限管理
- GitHub CLI のOAuthフローより劣る
- 手動トークン処理によるセキュリティリスクの増加

#### GitHub CLI が必須な理由

1. **セキュリティ**:
   - OAuthベースの認証（静的トークンより安全）
   - 自動トークンローテーションとリフレッシュ
   - 手動トークン作成や保存が不要
   - トークン漏洩リスクの削減

2. **利便性**:
   - 一度の `gh auth login` セットアップ
   - すべてのリポジトリで動作
   - トークンのコピー/ペースト不要
   - 自動認証情報管理

3. **ベストプラクティス**:
   - GitHubが推奨する認証方法
   - CLI自動化の業界標準
   - より良い監査証跡
   - チームオンボーディングが容易

#### セットアップ手順

1. **GitHub CLI をインストール**:
   ```bash
   # macOS
   brew install gh
   
   # Linux (Debian/Ubuntu)
   sudo apt install gh
   
   # Windows
   # ダウンロード: https://cli.github.com/
   ```

2. **認証**:
   ```bash
   gh auth login
   # ブラウザ経由でログインするための対話型プロンプトに従う
   ```

3. **認証の確認**:
   ```bash
   gh auth status
   # 表示: ✓ Logged in to github.com as YOUR_USERNAME
   ```

#### 実施

- スクリプトは認証にGitHub CLIを使用する必要がある
- スクリプトはGitHub CLIの存在を検出する必要がある: `gh auth status`
- GitHub CLIが認証されていない場合、明確なエラーでスクリプトは失敗する必要がある
- ドキュメントはユーザーにGitHub CLIの使用を指示する必要がある
- コードレビューはPATの使用を拒否する必要がある

**ルール**: 「GitHub認証には常にGitHub CLI (`gh`) を使用すること。Personal Access Token（ClassicまたはFine-grained）は、CI/CDワークフローにおけるGitHub Actions組み込みトークンを除き、厳格に禁止されています。」

## **📊 データモデル仕様**

### CloudService モデル（標準化されたサービス表現）
すべてのクラウドプロバイダーのリソースを以下の統一モデルにマッピング：

```python
@dataclass
class CloudService:
    provider: str        # "aws" | "gcp" | "azure"
    service_type: str    # "EC2", "Compute Engine", "Virtual Machine" など
    name: str            # リソース名またはID
    region: str          # リージョン名
    status: str          # ステータス（プロバイダー固有）
    created_at: str      # 作成日時（ISO 8601形式）
    metadata: dict       # プロバイダー固有の追加情報
```

### 出力フォーマット
- **table**: デフォルト、Rich ライブラリを使用した見やすいテーブル
- **json**: JSON配列形式
- **csv**: CSV形式（ヘッダー付き）

### コマンド未検出ルール（必須）
必要なコマンドが見つからない場合：

1. **必ずコマンドをインストールすること**:
   - コマンドがないから実行をスキップしないこと
   - 代替コマンドを探さないこと
   - 回避策を試みないこと

2. **インストールプロセス**:
   - コマンドの正しいパッケージ名を特定
   - 適切なパッケージマネージャーを使用してインストール:
     - Linux (Debian/Ubuntu): `apt-get update && apt-get install -y パッケージ名`
     - Linux (Fedora/RHEL): `yum install -y パッケージ名`
     - macOS: `brew install パッケージ名`
     - その他: システムの適切なパッケージマネージャーを使用
   - インストール確認: `which コマンド名` または `コマンド名 --version`
   - インストール成功後、元の操作を実行

3. **ドキュメント化の要件**:
   - コマンドをインストールした後、ドキュメント化すること:
     - `04_SETUP.md` にインストール手順を追加
     - コマンド名とインストール方法を記載
     - 検証ステップを追加
     - 英語版と日本語版の両方を更新

4. **シナリオの例**:
   - `git` コマンドが見つからない → git をインストール
   - `docker` コマンドが見つからない → docker をインストール
   - `python` コマンドが見つからない → python をインストール
   - `curl` コマンドが見つからない → curl をインストール

**ルール**: 「必要なコマンドがあれば、インストールすること。スキップや代替を試みないこと。」

### ディレクトリ操作ルール（必須）
ディレクトリに対して操作するとき：

1. **操作前に必ずディレクトリ構成を確認すること**:
   - ターゲットディレクトリをリスト: `ls -la <ターゲットディレクトリ>`
   - 現在のディレクトリを表示: `pwd`
   - 操作前に正確なパスを確認
   - ディレクトリ階層と内容を理解

2. **ディレクトリ確認チェックリスト**:
   - ✅ パスは正しいか（絶対パスか相対パスか）？
   - ✅ ディレクトリは存在するか？
   - ✅ 期待されるファイル/サブディレクトリが存在するか？
   - ✅ 適切なパーミッションがあるか？
   - ✅ 目的のディレクトリか（似た名前のものではないか）？

3. **破壊的な操作の前に**:
   - **必ず**ディレクトリの内容をリスト: `ls -R <ターゲットディレクトリ>`
   - **必ず**正しい位置を操作しているか確認
   - **必ず**ファイル/ディレクトリ名が期待値と一致することを確認
   - **決して**検証なしで削除操作を実行しないこと

4. **一般的な確認コマンド**:
   ```bash
   pwd                          # 現在のディレクトリを表示
   ls -la <ディレクトリ>         # ディレクトリ内容を詳細表示
   find <ディレクトリ> -type f   # ディレクトリ内のすべてのファイルを検索
   tree <ディレクトリ>           # ディレクトリツリーを表示（インストール必須）
   stat <ディレクトリ>           # ディレクトリ情報を表示
   ```

5. **操作フロー例**:
   ```bash
   # ディレクトリを削除する前に:
   pwd                          # 現在位置を確認
   ls -la /path/to/delete       # 内容を最初に確認
   ls -la /path/to/delete/*     # 中身を確認
   # 確認後に、削除が正しい場合のみ実行
   rm -rf /path/to/delete
   ```

**ルール**: 「操作前に常にディレクトリ構成を確認すること。一つの確認が一つの誤りを防ぐ。」

### 段階的開発ルール（必須）

ファイル作成、ドキュメント作成、または機能実装を行う際：

1. **ファイルは必ず段階的に作成すること（一度に1-3ファイル）**:
   - 1-3ファイルを作成したら停止し、確認を待つ
   - 一度に5ファイル以上を作成しない
   - ネットワークタイムアウトやエラーを防止
   - レビューと方向修正を可能にする

2. **ファイル作成戦略**:
   - ✅ **優先度1**: 可能な限り既存ファイルを更新
   - ✅ **優先度2**: 必須ファイルのみ作成
   - ✅ **優先度3**: オプションファイルについてユーザーに確認
   - ✅ **優先度4**: 残りのファイルを確認を取りながら段階的に作成

3. **停止して確認すべきタイミング**:
   - 2-3ファイル作成後
   - 複雑な操作や大規模なコード変更の後
   - 合計出力が約500行を超えた時
   - 次の主要タスクや機能に進む前

4. **常にユーザーに確認**:
   ```
   ✅ 作成完了（Xファイル）:
      - file1.py
      - file2.md
   
   📋 残りの作業:
      - file3.py
      - file4.md
      - file5.json
   
   次の2-3ファイルを作成しますか？
   ```

5. **例 - 良い手順**:
   ```
   ステップ1: コア実装を作成（1-2ファイル）
   [ユーザー確認を待つ] ✅
   
   ステップ2: ドキュメント作成（1-2ファイル）
   [ユーザー確認を待つ] ✅
   
   ステップ3: サンプルとテストを作成（1-2ファイル）
   [ユーザー確認を待つ] ✅
   ```

6. **例 - 悪い手順**（実行しないこと）:
   ```
   一度に10ファイルを作成:
   - file1.py, file2.py, file3.py, file4.py, file5.py
   - doc1.md, doc2.md, doc3.md
   - test1.py, test2.py
   ❌ ネットワークタイムアウト発生
   ❌ 部分的完了、不明確な状態
   ❌ レビューが困難
   ```

**根拠**: ネットワーク制約と安定性。一度に多数のファイル作成や大量出力を生成すると、ネットワークタイムアウトやエラーが発生します。段階的作成により、安定した動作と反復的なフィードバックが可能になります。

**ルール**: 「段階的に作成し、頻繁に確認。一度に1-3ファイルでエラーを回避。」

  - QA 検査済みで本番環境対応可能
  - 保護されたブランチ（直接コミット不可）
  
- **`develop` ブランチ**: 開発統合ブランチ
  - 最新の開発変更を含む
  - すべての機能統合ポイント
  - 安定性とテスト済みが必須
  - 保護されたブランチ（PR 経由でのマージのみ）

- **機能/バグフィックスブランチ**: 開発作業用ブランチ
  - フォーマット: `feature/機能名` または `bugfix/バグ名`
  - 作成元: `develop` ブランチ
  - マージ先: `develop` ブランチ
  - マージ後に削除

#### 開発ワークフロー（必須）

1. **作業開始時**:
   - `develop` からフィーチャー/バグフィックスブランチを作成
   - ブランチ命名: `feature/説明的な名前` または `bugfix/イシュー番号`
   - 例: `feature/list-services-gcp`, `bugfix/aws-auth-error`

2. **開発中**:
   - コミットはアトミック（原子的）に、わかりやすく説明
   - 頻繁にコミット（明確なメッセージ付き）
   - コミット前にすべてのテストが成功していること

3. **作業完了時**:
   - ローカルですべてのテストが成功することを確認: `pytest`
   - リモートにブランチをプッシュ
   - `develop` へのプルリクエスト（PR）を作成
   - コード審査フィードバックに対応

4. **develop へのマージ**:
   - PR は少なくとも 1 人のレビュアーに承認される必要がある
   - すべての CI/CD チェックに合格する必要がある
   - 必要に応じてコミットをスカッシュ（履歴をクリーン化）
   - マージ後、フィーチャーブランチを削除

5. **main へのリリース**:
   - リリース時のみ `develop` → `main` へマージ
   - `develop` のすべてのテストが合格していることを確認
   - `develop` から `main` へのリリース PR を作成
   - バージョン番号付きでタグ付け（例: `v1.0.0`）
   - リリースノートを作成

#### コミットメッセージ標準
- **フォーマット**: `Type: 寧ろな説明`
- **タイプ**: `feat`, `fix`, `docs`, `test`, `refactor`, `chore`
- **例**:
  - `feat: GCP プロバイダーサポートを追加`
  - `fix: AWS 認証タイムアウトの問題を解決`
  - `docs: API 設計仕様書を更新`
  - `test: Azure プロバイダーのユニットテストを追加`

#### QA/テスト要件（必須）

`develop` または `main` へマージする前に:
- ✅ すべてのユニットテストが成功: `pytest --cov=src`
- ✅ コードカバレッジが 80% 以上に維持
- ✅ 認証情報がハードコードされていない
- ✅ すべての新しい関数に型ヒント
- ✅ すべてのパブリックメソッドに docstring
- ✅ ドキュメントが更新されていること（ドキュメントポリシー参照）
- ✅ リント成功: `ruff check src/`
- ✅ コードフォーマット完了: `black src/`

`main` へマージする前に:
- ✅ `develop` ブランチが完全にテスト済みで安定している
- ✅ リリースをブロックする未解決の問題がない
- ✅ リリースノートが準備されている
- ✅ バージョン番号が更新されている

#### ルール要約
> \"main または develop へ直接コミットしてはいけない。常に develop からフィーチャーブランチで作業すること。develop へのマージ前に品質を確認し、main へのリリース前に安定性を確認すること。\"
### バージョニング戦略

#### バージョン番号形式
**形式**: `W.X.Y.Z`（`1.0.0.0` から開始）

#### バージョン番号を増やすルール

| コンポーネント | 増やすタイミング | トリガー | 例 |
|----------|-----------|--------|-----|
| **W** (メジャー) | ユーザーが新しいメジャー指示/要件を与えたとき | ユーザーのリクエスト/指示 | `1.5.10.15` → `2.0.10.15` |
| **X** (マイナー) | ユーザーが新しい機能/改善指示を与えたとき | ユーザーのリクエスト/指示 | `1.0.5.10` → `1.1.5.10` |
| **Y** (開発版) | `develop` ブランチに変更をプッシュしたとき | `git push origin develop` | `1.0.5.10` → `1.0.6.10` |
| **Z** (コミット) | 新しいコミットを作成したとき | `git commit ...` | `1.0.5.10` → `1.0.5.11` |

#### バージョニングルール（必須）

1. **開始バージョン**: `1.0.0.0`（プロジェクト開始時）

2. **メジャーバージョン (W) の増加**:
   - ユーザーが重大な新しい指示または根本的な変更を与えたときにトリガー
   - 例: フェーズ遷移、メジャーアーキテクチャ再設計、完全な機能セット再設計
   - W を増加し、X を 0 にリセット（例: `1.15.10.7` → `2.0.10.7`）
   - Y と Z は独立して継続増加

3. **マイナーバージョン (X) の増加**:
   - ユーザーが新しい機能指示または重要な改善指示を与えたときにトリガー
   - 例: 新しい CLI コマンド、新しいプロバイダーサポート、新しいワークフロー
   - X のみ増加（例: `1.0.5.3` → `1.1.5.3`）
   - Y と Z は独立して継続増加

4. **開発バージョン (Y) の増加**:
   - `develop` ブランチに変更がプッシュされるたび、毎回増加
   - プルリクエスト `develop` へのマージ成功後に適用
   - 開発サイクルの進捗を示す
   - Z は独立して継続増加（例: `1.0.0.5` → `1.0.1.5`）

5. **コミットバージョン (Z) の増加**:
   - 新しいコミットが作成されるたび、毎回増加
   - 実行時に適用: `git commit ...`
   - きめ細かい開発進捗を追跡
   - リセットなし、継続増加（例: `1.0.0.5` → `1.0.0.6`）

#### コード内でのバージョン管理

1. **バージョンの保存場所**: `config.py`
   ```python
   VERSION = "1.0.0.0"
   ```

2. **コミット時のバージョン更新**:
   - コード提出前に `config.py` の VERSION を更新
   - Z コンポーネントを増加
   - コミットメッセージに含める: `chore: bump version to 1.0.0.X`

3. **develop プッシュ時のバージョン更新**:
   - PR を `develop` にマージした後、Y コンポーネントを増加
   - 別コミット作成: `chore: bump dev version to 1.0.X.0`
   - `05_DEVELOPMENT_CHECKLIST.md` を現在のバージョンで更新
   - 後続のコミット時に Z は通常通り継続増加

4. **メジャー/マイナー更新**:
   - ユーザーが W または X を増やすための指示を与えたとき
   - 直ちにバージョン更新を含むコミットを作成
   - `05_DEVELOPMENT_CHECKLIST.md` とすべてのドキュメントを更新

#### CLI でのバージョン表示

- 以下で バージョンを表示: `python -m src.cli.main --version`
- main.py に実装:
  ```python
  from config import VERSION
  
  @app.command()
  def version():
      """アプリケーションバージョンを表示。"""
      print(f"Cloud Service Manager v{VERSION}")
  ```

#### ルール要約
> \"バージョン番号は開発ペースを反映している: W 増加時は X をリセット（0に戻す）、X は独立して増加、Y は独立して開発統合サイクルを追跡、Z は独立してすべてのアトミックコミットを追跡する。バージョンは常にコード内で最新である必要がある。\"
