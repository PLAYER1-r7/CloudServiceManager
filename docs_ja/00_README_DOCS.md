# ドキュメント読み取りガイド / Documentation Reading Guide

## **AIエージェントへの指示 / Instructions for AI Agents**

このプロジェクトで作業を行う前に、**必ず以下の順序でドキュメントを読んでください**。

### 📖 必須読み取り順序 (Required Reading Order)

1. **[01_PREREQUISITES.md](01_PREREQUISITES.md)** ⚠️ **最優先・必読**
   - プロジェクトの前提条件、制約、重要な技術的決定事項

2. **[02_PROJECT_PLAN.md](02_PROJECT_PLAN.md)**
   - プロジェクト全体の概要、目的、フェーズ、技術スタック

3. **[03_API_DESIGN.md](03_API_DESIGN.md)**
   - CLI コマンドの仕様、API 設計
   - 入出力形式、データモデル

4. **[04_SETUP.md](04_SETUP.md)**
   - 開発環境のセットアップ手順
   - クラウドプロバイダー認証情報の設定

5. **[05_DEVELOPMENT_CHECKLIST.md](05_DEVELOPMENT_CHECKLIST.md)**
   - 開発セットアップの進捗状況
   - 次に実装すべき機能

## **ドキュメント更新時の注意 / Document Update Guidelines**

### AIエージェント向けガイドライン

- **構造化**: 各ドキュメントはマークダウンで構造化し、セクションを明確に分ける
- **明示的**: 暗黙の前提を避け、すべての制約と決定を明示的に記載する
- **コンテキスト**: 各ドキュメントの先頭に「目的」「対象読者」「前提知識」を記載
- **一貫性**: 用語、命名規則、フォーマットをドキュメント間で統一
- **更新日**: 重要な変更を行った場合は更新日を記録
- **マルチ言語**: `/docs` は英語、`/docs_ja` は日本語で統一して記載すること
- **コマンド実行**: 外部コマンドを実行する前に必ず使用方法を確認。誤ったUsageで失敗した場合は、正しいコマンドを関連ドキュメントに記録すること
- **Wiki同期**: ユーザー向けの使い方・セットアップ・ワークフロー・トラブルシュートに影響する変更を行った場合は、`/docs` と `/docs_ja` に加えて GitHub Wiki も更新すること

### ディレクトリ責務分離ポリシー（必須）

ドキュメントの発見性を高め、重複を防ぐため、本プロジェクトでは責務で配置先を分離する:

- **`/docs` と `/docs_ja`**: プロダクト実装ドキュメントのみ
   - アーキテクチャ、API/CLI仕様、セットアップ手順、開発制約、チェックリスト
   - 答えるべき問い: 「プロダクトをどう作る・動かす・保守するか」

- **`/.github`**: リポジトリ運用・GitHubワークフロードキュメントのみ
   - GitHub Projects運用、Issue/PRフロー、ブランチ保護、自動化スクリプト、エージェント運用ガイド
   - 答えるべき問い: 「GitHub上で作業をどう管理・運用するか」

ルール:
- プロダクト実装仕様を `/.github` に移さないこと。
- GitHub運用手順を `/docs` や `/docs_ja` に置かないこと。
- 両方の領域にまたがる内容は、主責務側に原本を置き、もう一方はリンク参照にすること。
- 新規ドキュメント追加時は、言語より先に目的（責務）で配置先を決めること。

### 外部コマンド実行ルール

**外部コマンド（CLI、API、スクリプト等）を実行する際の必須ルール:**

1. **必ず最初に使用方法を確認**:
   - 実行前に `command --help` または `command -h` を実行
   - 公式ドキュメントがあれば参照
   - 必須引数とオプションをすべて確認

2. **使用方法の間違いが発生した場合**:
   - コマンドが誤ったUsageで失敗した場合
   - 修正して正しく実行できた後
   - **正しい使用方法を必ずドキュメント化する** 以下の場所に:
     - コードコメント（スクリプト内のコマンドの場合）
     - `04_SETUP.md`（セットアップ/環境構築コマンドの場合）
     - `05_DEVELOPMENT_CHECKLIST.md`（開発コマンドの場合）
     - 関連する技術ドキュメント

3. **ドキュメント形式**:
   ```markdown
   ### コマンド: <コマンド名>
   **目的**: 簡潔な説明  
   **使用方法**: command [OPTIONS] [ARGUMENTS]
   **例**: actual-working-command --option value
   **注意事項**: 重要な注意点または一般的な落とし穴
   ```

**ルール**: 「失敗から学ぶ - 修正したコマンドを将来の参考のためにドキュメント化する」

### 読み取りパターン

```
作業開始時:
  1. 01_PREREQUISITES.md を必ず読む
  2. タスクに応じて必要なドキュメントを参照
  
プロジェクト管理を開始する際（AIエージェント）:
  1. ../.github/AI_AGENT_PROJECT_GUIDE.md を読む
  2. 実行: python .github/project_manager.py all
  3. 推奨タスクと依存関係を確認
  4. ブロッカーがない最高優先順位のタスクを選択
  
新機能実装時:
  1. 01_PREREQUISITES.md で制約を確認
  2. 02_PROJECT_PLAN.md でアーキテクチャを確認
  3. 03_API_DESIGN.md で仕様を確認
  4. 実装
  5. テストを記述
  6. ドキュメントを更新（必須 - 以下参照）
  7. 05_DEVELOPMENT_CHECKLIST.md を更新
  
環境セットアップ時:
  1. 01_PREREQUISITES.md で要件を確認
  2. 04_SETUP.md の手順に従う
  3. 05_DEVELOPMENT_CHECKLIST.md でチェック
```

## **ドキュメント言語ポリシー**

### 言語ルール（必須）
- **`/docs`**: すべてのドキュメントは**英語**で記述（AIエージェント向けに最適化）
- **`/docs_ja`**: すべてのドキュメントは**日本語**で記述（人間の開発者向け）
- **言語混在禁止**: 単一のドキュメント内で言語を混在させないこと
- **同期必須**: ドキュメント更新時は、英語版と日本語版の両方を更新すること

### 更新ルール（必須）
**いかなる作業を完了した後も、必ずドキュメントを更新すること:**
- ✅ 常に `/docs`（英語）を最初に更新
- ✅ 次にすぐに `/docs_ja`（日本語）を更新
- ✅ ユーザー向け内容に変更がある場合は、GitHub Wiki（例: Home / Getting-Started / FAQ）も更新
- ✅ 更新はオプションではなく要件
- ✅ 詳細な更新プロセスは [01_PREREQUISITES.md](01_PREREQUISITES.md) を参照

**ルール**: 「ドキュメントを更新するまで作業は完了していない」
